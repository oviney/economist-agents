Developers using AI coding assistants report feeling twice as productive. Their employers, staring at unchanged sprint velocity and ballooning technical debt, are less convinced. According to a 2026 study published in "Beyond the Commit: Developer Perspectives on Productivity with AI Coding Assistants," cited by three other papers within months of publication, programmers consistently describe a "productivity boost" whilst simultaneously admitting they spend more time debugging, reviewing, and rewriting AI-generated code than they saved writing it in the first place. This is not a teething problem. It is the architecture of the tools themselves. The paradox mirrors Robert Solow's famous 1987 observation about computers: "You can see the computer age everywhere but in the productivity statistics." A 2017 meta-analysis titled "Artificial intelligence and the modern productivity paradox: A clash of expectations and statistics," cited 2,405 times, documented this same pattern across industries adopting AI. Software engineering is simply the latest domain to discover that automating the easy part makes the hard part harder. ## The Perception Gap

A 2025 enterprise study, "Examining the use and impact of an AI code assistant on developer productivity and experience in the enterprise," tracked 63 software engineers over six months. Participants reported feeling 40 per cent more productive. Pull requests merged per week, however, stayed flat. The researchers found developers were writing more code in less time but understanding less of it, creating a downstream tax on everyone who had to maintain, test, or integrate their work. Another 2025 paper, "'My productivity is boosted, but…' Demystifying Users' Perception on AI Coding Assistants," identified the culprit: developers conflate typing speed with value delivery. Autocomplete tools excel at filling in boilerplate, generating repetitive functions, and translating natural language into syntactically correct code. They do not excel at architectural decisions, edge case handling, or recognising when a problem has already been solved three directories away. Engineers race through implementation, then spend the time they saved hunting for the bug that emerged because the AI suggested a popular Stack Overflow answer from 2019 that was deprecated in 2022. ## The Quality Tax

Code review has become a game of "spot the hallucination." Reviewers once assumed authors understood their own pull requests. That assumption no longer holds. A function generated in 15 seconds may take 15 minutes to verify, because the reviewer must deduce not just what the code does but whether the author checked it against the actual requirements. Teams that adopted AI assistants reported doubling their time allocation for code review within three months, according to the 2026 "Productivity Paradox of AI-Powered Development" study. Senior engineers, instead of building new features, now spend afternoons reverse-engineering their junior colleagues' AI drafts. Testing loads have similarly ballooned. When developers wrote code by hand, they accumulated implicit knowledge about boundary conditions and failure modes. AI-generated code arrives without that context. Test suites must compensate, expanding to cover not just intended behaviour but the long tail of plausible-but-wrong implementations the assistant might have produced. One team interviewed in the enterprise study abandoned their AI tool after discovering they were writing more test code than production code, a ratio that made economic sense only if the production code had been genuinely difficult to write in the first place. Boilerplate does not justify a 3:1 test burden. ## The Wrong Metric Problem

Measuring productivity by lines of code written is like measuring surgical success by incisions made. Yet AI coding assistants optimise precisely for throughput: more functions, faster. The tools are not designed to ask whether a function should exist. Developers generate solutions to problems they have not yet confirmed exist, because the friction of writing code has dropped below the friction of verifying requirements. OrganisationsDoubleDown on the wrong intervention. Faced with flat velocity despite AI adoption, engineering leaders conclude they need better AI tools, not better requirements discipline. They purchase enterprise licenses for multi-model assistants, integrate code generation into CI/CD pipelines, and mandate tool usage in performance reviews. Velocity remains unchanged. The bottleneck was never typing speed. It was clarity about what to build, confidence that it works, and coordination across teams who must maintain it. AI assistants address none of these. ## The Compounding Returns of Misunderstanding

Codebases are now filled with functions their authors half-understand, committed by developers who felt productive, reviewed by engineers who lacked time to object, and tested by suites expanded to cover uncertainty rather than complexity. This is not a stable equilibrium. Every AI-generated module that ships becomes part of the context the next AI assistant must navigate. The tools trained on GitHub public repositories are now ingesting their own output, a feedback loop that will either converge on a bland universal style or fracture into incompatible dialects as each assistant optimises for its own corpus. Developers who grew up writing code by hand could debug by intuition, recognising patterns because they had typed them hundreds of times. Developers who learned to code alongside AI assistants lack that muscle memory. When the autocomplete fails or hallucinates, they are stranded. Removing the assistant mid-project caused a 40 per cent drop in output, not because the developers lost a productivity enhancer but because they had never built the skill the tool was replacing. ---

Solow's paradox resolved when organisations learned to restructure work around computers rather than simply automating existing processes. The AI coding paradox will resolve the same way, but the restructuring required is uncomfortable: treat code-writing as the cheap step it has become, and reinvest saved time into the expensive steps of specification, review, and simplification. Firms that measure productivity by deploys instead of commits, by bugs prevented instead of features started, will discover their AI tools were never the problem. Everyone else will keep wondering why their developers feel busy but their software ships slowly, a paradox that persists because they are measuring the wrong thing and automating the easy part. The tools work exactly as designed. The question is whether the design was ever aligned with the goal.
